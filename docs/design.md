# 产品设计与决策记录

## 目标理解

- 构建一个浏览器端 Markdown 思维导图工具。
- Markdown 是唯一数据源，避免文档与脑图状态分裂。
- 支持粘贴、编辑和导入 Markdown 文件。
- Markdown 修改后实时生成可缩放、可折叠的思维导图。
- AI 作为增强能力，将普通文本提炼为结构化 Markdown。
- 首版不包含账号、云同步和多人协作。

## 非功能假设

- 本地优先；未主动调用 AI 时，内容不离开设备。
- 首版以单人使用为主。
- 普通电脑上约 1,000 个节点仍应保持基本流畅。
- 编辑内容需要自动保存，并提供明确的恢复和错误提示。
- 优先选择 MIT 或 Apache-2.0 依赖，降低后续维护和商业化限制。
- 项目由小团队长期维护，保持依赖数量与架构复杂度可控。

## 最终路线

- React + TypeScript + Vite 构建产品外壳。
- Markmap 作为 Markdown 脑图渲染引擎。
- Markdown 文本是应用状态的唯一真源。
- 普通导入、编辑、渲染和导出全部在浏览器完成。
- AI 请求通过 Vercel Function 转发，服务端密钥不暴露给浏览器。
- 界面采用本地优先的双栏工作台，窄屏时改为纵向布局。

## 决策记录

### D-001：自行开发产品外壳

- 选择：自行开发，复用成熟开源组件。
- 备选：直接 Fork myMarkmap；基于 Mind Elixir 开发。
- 原因：代码边界更清晰，中文体验、AI 能力和 Vercel 部署更容易按需设计。

### D-002：Markdown 为唯一数据源

- 选择：所有脑图内容由 Markdown 派生。
- 备选：脑图 JSON 与 Markdown 双向同步。
- 原因：首版更可靠，也更符合用户主要在 Markdown 模式修改的习惯。

### D-003：部署目标为 Vercel

- 选择：静态前端加可选的 Vercel Function。
- 备选：自建常驻后端。
- 原因：首版无需数据库或常驻服务，部署和预览流程更轻。

### D-004：未完成功能保持禁用

- 选择：仅将尚未完成的功能按钮明确禁用；功能完成后再开放。
- 备选：展示可点击但无行为的按钮。
- 原因：避免界面暗示不存在的功能，并为后续测试提供清晰边界；当前导入、导出和 AI 提炼均已开放。

### D-005：首版采用原生文本编辑器

- 选择：使用受控 `textarea` 作为 Markdown 唯一编辑入口。
- 备选：首版直接接入 CodeMirror。
- 原因：原生编辑器启动快、粘贴和键盘支持稳定、无额外依赖；后续需要语法高亮时仍可替换，不影响 Markdown 数据模型。

### D-006：导入限制为 5 MB

- 选择：仅接受 `.md`、`.markdown`，单文件不超过 5 MB。
- 备选：不限制文件大小。
- 原因：避免浏览器因异常大文件阻塞，同时覆盖远超普通脑图需求的文本规模。

### D-007：接入 Markmap 官方基础转换入口

- 选择：使用 `markmap-lib@0.18.12/no-plugins` 转换 Markdown，使用 `markmap-view@0.18.12` 渲染 SVG 脑图。
- 备选：使用包含数学公式和代码高亮插件的默认入口；自行实现脑图布局。
- 原因：官方基础入口已覆盖首版需要的标题与列表结构，同时避免把 KaTeX、Prism 等暂未使用的依赖打入首屏代码。

### D-008：脑图视图按需加载

- 选择：编辑器与 Markdown 转换器正常加载，交互式脑图视图在画布就绪后异步加载。
- 备选：所有能力一次性打入首屏脚本。
- 原因：保留即时文本反馈，并降低初始加载成本；生产构建不再产生超大代码块警告。

### D-009：版本化本地草稿

- 选择：将 Markdown 与文件名以版本化 JSON 保存在浏览器 `localStorage`，编辑停止 300ms 后写入。
- 备选：每次按键同步写入；首版直接接入数据库。
- 原因：防抖可以减少大文档编辑时的主线程开销，本地存储符合首版本地优先原则，也不需要账号体系。
- 异常处理：无效或旧格式草稿会被清理并回退默认内容；存储空间不足时保留当前编辑内容并显示错误提示。

### D-010：浏览器本地多格式导出

- 选择：使用浏览器原生 Blob、SVG 序列化和 Canvas 完成 Markdown、SVG、2× PNG 导出；PDF 使用按需加载的 PDF 生成器，将高清脑图缩放居中到 A4 横向单页。
- 备选：引入完整的 DOM 截图库；通过服务端生成文件；只提供浏览器打印对话框。
- 原因：原生导出依赖少、内容不离开设备；PDF 复用 PNG 渲染结果可以保留中文外观，无需额外嵌入中文字体，同时满足一键下载。
- 文件名：沿用当前 Markdown 文件的基础名称，清理文件系统不安全字符后替换扩展名。
- 边界：Markdown 始终可导出；空文档时禁用 SVG、PNG 和 PDF；首版 PDF 为单页 A4 横向，超大脑图会整体缩小而不分页。
- 反馈与测试：导出成功或失败显示在工作区底部；自动测试覆盖菜单、格式和文件名，真实浏览器验证四种下载及响应式布局。

### D-011：豆包优先的可扩展 AI Provider

- 目标：将普通文本提炼为层级清晰的 Markdown，用户预览并确认后才替换编辑器内容。
- 选择：首个 Provider 使用火山方舟豆包，通过 Vercel Function 在服务端调用 `POST /api/v3/chat/completions`；前端只请求站内 `/api/refine`。
- 模型：默认 `doubao-seed-2-1-pro-260628`，可通过服务端 `ARK_MODEL_ID` 环境变量替换；密钥仅从 `ARK_API_KEY` 读取。
- 调用方式：首版使用非流式请求，完整 Markdown 返回后展示预览，避免半截内容进入编辑器。
- 架构：定义统一 Provider 接口；豆包实现使用原生 `fetch`，后续新增其他模型时不改动前端协议。
- 备选：浏览器直连方舟；在 Function 中安装官方 SDK；为每个模型单独编写前端流程。
- 原因：服务端代理不会暴露 API Key；原生 HTTP 调用依赖更少；统一 Provider 能控制后续多模型扩展成本。
- 输入边界：仅接受 1 至 20,000 个字符；输出只接受非空 Markdown；请求和响应均不缓存，不记录用户原文。
- 可靠性：上游请求设置超时；服务端隐藏上游错误细节，前端提供重试；缺少密钥时返回明确的未配置状态。
- 交互：智能提炼按钮打开对话框，包含原文输入、隐私提示、加载状态、Markdown 预览、重新编辑、取消和确认应用。
- 非目标：首版不做流式输出、多模型切换器、聊天历史、账号计费、云端保存和自动覆盖编辑器。
- 测试：Provider 与 API 使用模拟上游请求，界面测试覆盖打开、校验、加载、错误、预览和确认应用；真实密钥仅在部署环境中验收。
